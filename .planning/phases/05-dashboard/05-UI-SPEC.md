**Accent reserved for (Phase 5):** Nothing new. `bg-primary` is not introduced anywhere in this phase's new
UI surface — the dashboard is read-only (no create/save/confirm actions), and the low-stock banner's
"View all products" link/button uses `variant="ghost"` or `variant="outline"` (a neutral, non-primary
treatment), consistent with the 10% accent discipline: **primary is reserved exclusively for the single
highest-intent mutating action on a page**, and this phase has none. This is a deliberate absence, not an
oversight — do not add a `bg-primary` button to `/dashboard` to "fill" the accent slot.

### Severity/Semantic Color Reuse (inherited from Phase 2 — NOT the 10% accent slot)

The low-stock KPI tile's icon uses the same amber "Warning" semantic tone already established for severity
badges (`lib/utils/severity.ts`) — this is a semantic/status color, not the primary-accent token, and does
not compete with or dilute the 10% accent reservation above.

| Element | Color | Rationale |
|---------|-------|-----------|
| Low-Stock KPI tile icon (`AlertTriangle`) | `text-amber-600` | Matches the amber family already used for `getSeverityBadge`'s "Warning" tier — signals "needs attention" at a glance, consistent with the existing severity color vocabulary |
| Other 3 KPI tile icons (`Package`, `Truck`, `Activity`) | `text-muted-foreground` | Neutral, informational tiles — no attention-seeking treatment needed |

### PO Status Pie Chart Colors (Phase 5 new — Claude's Discretion per CONTEXT.md, resolved here)

Recharts requires solid hex fill values (CSS variables/oklch are not reliably supported by SVG `fill`
across all Recharts render paths), so the pie chart uses mid-tone hex equivalents of the same three status
families already established by `getStatusBadge` (`lib/utils/po-status.ts`) — same family, different
context (solid fill vs. tinted badge background):

| Status | Badge (existing, tinted) | Pie slice fill (new, solid) | Family |
|--------|---------------------------|-------------------------------|--------|
| DRAFT | `bg-slate-100 text-slate-700` | `#64748b` (slate-500) | Neutral/slate |
| ORDERED | `bg-blue-100 text-blue-700` | `#3b82f6` (blue-500) | Blue |
| RECEIVED | `bg-green-100 text-green-700` | `#22c55e` (green-500) | Green |

```typescript
const STATUS_COLORS = { DRAFT: "#64748b", ORDERED: "#3b82f6", RECEIVED: "#22c55e" }
```

This keeps the pie chart visually traceable to the same status badges used everywhere else in the app
(`/purchase-orders` list, PO detail page) — a manager should not need to learn a second color vocabulary for
the same three statuses.

### Currency Display

Not applicable — Phase 5's PO status aggregation is counts only (`groupBy(['status'], {_count})`), never
`totalAmount` (RESEARCH.md Pitfall 3). No currency formatting is introduced or needed in this phase.

**PO status pie chart — empty state (Claude's Discretion, resolved here):** If `PurchaseOrder.count() === 0`
(no POs exist anywhere in the system yet, e.g. a fresh install before Phase 4 data exists), do NOT render an
all-zero `<PieChart>` — Recharts renders a blank/degenerate arc for an all-zero dataset, which reads as a
broken chart, not an empty state. Instead render a simple text empty state inside the same Card:
```tsx
<div className="flex flex-col items-center py-12 text-center">
  <ClipboardList className="w-8 h-8 text-muted-foreground/30 mb-3" />
  <p className="text-sm font-medium">No purchase orders yet</p>

  </p>
</div>
```

structure) for cross-page consistency. If at least one PO exists in any status, render the full `<PieChart>`
even if one or two of the three statuses are currently at 0 (Recharts omits/minimizes zero-value slices
natively — no special handling needed for the "some but not all statuses populated" case).

**KPI tiles never show a dedicated "empty" treatment for zero values** (e.g. `0` active suppliers) — a plain
"0" is a valid, informative count on this dashboard, not an empty/error state. Only the pie chart (a visual,
not a number) needs the empty-state substitution above.

### Screen 3: Purchase Orders List Page — Status Drill-Down Extension (`/purchase-orders?status={STATUS}`)

**Access:** Both MANAGER and STAFF (unchanged, D-14 from Phase 4). This is an additive extension to
04-UI-SPEC's Screen 1 — the page's layout, table columns, and empty states are UNCHANGED. The new work is
purely the URL-param entry point into the existing Tabs filter (RESEARCH.md Pitfall 1: this did NOT
previously exist — `purchase-orders-client.tsx` only had local `useState<FilterTab>("all")` with no
`searchParams` read at all; CONTEXT.md's phrasing that this "already exists" is corrected here).

**What's new this phase:**
1. `page.tsx` adds `searchParams: Promise<{status?: string}>`, awaits it, validates the value against the
   `POStatus` enum (`"DRAFT" | "ORDERED" | "RECEIVED"`, case-sensitive uppercase to match the dashboard's
   `router.push` value) — any other value or absence falls back to `undefined`/no filter, never a thrown
   error (RESEARCH.md Pitfall 2).
2. `page.tsx` passes the validated value (lowercased to match `FilterTab`'s existing `"draft"|"ordered"
   |"received"|"all"` type) as an `initialFilter` prop.
3. `purchase-orders-client.tsx`'s `useState<FilterTab>("all")` becomes `useState<FilterTab>(initialFilter ??
   "all")` — a one-line seed change, Tabs UI itself is unchanged.

**No new visual elements** — clicking a pie slice on `/dashboard` simply lands the user on the exact same
`/purchase-orders` page they'd reach by manually clicking a Tab, just pre-selected. No banner is needed here
(unlike Screen 2) because the Tabs UI itself already visually communicates the active filter (the selected
Tab is highlighted) — D-05's banner requirement was specific to `/products`, which had no equivalent
always-visible filter-state indicator before this phase.

## Interaction Contracts

### Dashboard data fetch (DASH-01/02/03)

1. `page.tsx` (Server Component) runs 5 Prisma queries in parallel via `Promise.all`: `product.count`,
   `supplier.count`, `stockTransaction.count` (UTC-day-bounded, D-10), `product.count` with the FieldRef
   low-stock filter (D-03/D-04), and `purchaseOrder.groupBy(['status'], {_count})`.
2. `groupBy` omits statuses with zero POs — `page.tsx` must default-fill all three statuses to 0 before
   passing to the client (RESEARCH.md Code Examples — `poStatusCounts` object pattern).
3. All values passed to `DashboardClient` are plain numbers — no `Decimal`, no Prisma model instances cross
   the Server→Client boundary (RESEARCH.md Pitfall 3: the `groupBy` query never touches `totalAmount`).

### Low-stock drill-down (DASH-02)

1. User clicks the Low Stock KPI tile → `<Link href="/products?stock=low">` — a real navigation, not a
   client-side-only state change.
2. `/products/page.tsx` reads `searchParams.stock`; if exactly `"low"`, applies
   `currentStock: { lte: prisma.product.fields.reorderThreshold }` AND `isActive: true` to the `findMany`
   query (D-01–D-04).
3. `products-client.tsx` receives an `isLowStockFiltered: boolean` + `lowStockCount: number` prop pair and
   conditionally renders the banner (D-05) above the table.
4. "View all products" navigates to `/products` (no query string) — a plain link, not a client-side filter
   reset, since the filtering happens server-side.

### PO status drill-down (DASH-03)

1. User clicks a pie slice → `onClick` handler (Recharts `<Pie onClick={...}>`) reads the clicked entry's
   `status` field and calls `useRouter().push(`/purchase-orders?status=${status}`)` (client-side navigation
   via `next/navigation`).
2. `/purchase-orders/page.tsx` reads `searchParams.status`; validates against the `POStatus` union
   (`"DRAFT"|"ORDERED"|"RECEIVED"`); invalid/absent → `undefined`.
3. Validated value is lowercased and passed as `initialFilter` to `purchase-orders-client.tsx`, which seeds
   `useState<FilterTab>(initialFilter ?? "all")` — a one-line change, all other Tabs behavior unchanged.

### Query-param validation (RESEARCH.md Pitfall 2 — hard requirement, both new params)

Both `?stock=` and `?status=` MUST use whitelist validation, never pass an unvalidated string into a Prisma
`where` clause or a `FilterTab`/`POStatus`-typed variable:
- `?stock=`: only the exact literal `"low"` triggers the filter; anything else (including `"LOW"`,
  `"true"`, empty string, missing) → unfiltered default, no error.
- `?status=`: only `"DRAFT"`, `"ORDERED"`, `"RECEIVED"` (case-sensitive, matching the Prisma enum and what
  the dashboard's `router.push` will always send) are accepted; anything else → `"all"` tab, no error. This
  closes the same class of gap already tracked as open technical debt on `/inventory` (STATE.md T-03-11) —
  do not repeat it here.

### Date/"today" boundary (D-10, restated for executor convenience)

"Movements today" uses UTC calendar-day boundaries, identical to the `/inventory` date-filter convention
(03-CONTEXT.md), NOT local server time and NOT a rolling 24-hour window:
```typescript
const now = new Date()
const todayStart = new Date(Date.UTC(now.getUTCFullYear(), now.getUTCMonth(), now.getUTCDate(), 0, 0, 0, 0))
const todayEnd = new Date(Date.UTC(now.getUTCFullYear(), now.getUTCMonth(), now.getUTCDate(), 23, 59, 59, 999))
```
