<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser URL bar → `/products?stock=` searchParams | Any string value is attacker/user-controllable and reaches the Server Component as raw text (session auth is already required to reach the route at all, but the *value* of `stock` is untrusted) |
| `page.tsx` → Prisma `where` clause | Only a validated boolean (`isLowStockFiltered`) branches into the FieldRef `where` object — the raw string itself never reaches Prisma |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-03 | Denial of Service | `app/(protected)/products/page.tsx` `searchParams.stock` | high | mitigate | Whitelist-validate: only the exact literal `"low"` sets `isLowStockFiltered = true`; every other value (different casing, empty string, arbitrary text, absent) leaves the `where` clause at `{}` — never passed to `new Date()` or any parser that could throw. This closes the same class of gap already tracked as open technical debt on `/inventory` (STATE.md T-03-11); this plan must not repeat it. |
| T-05-04 | Information Disclosure | `/products?stock=low` filtered list | low | accept | No new fields or access exposed — `/products` is already visible to both MANAGER and STAFF (AUTH-03); the filter only narrows rows already visible in the unfiltered view. |

</threat_model>
