## Interaction Contracts

### Report-type switching (D-01/D-02/D-03)

1. `page.tsx` (Server Component) reads `searchParams.type`, whitelist-validates against
   `["inventory", "movements", "purchase-orders"]`; any other value or absence resolves to `"inventory"`,
   never throws.
2. Only the active tab's Prisma query runs (D-03) — the page does not fetch all three reports in parallel.
3. `reports-client.tsx`'s `<Tabs>` `onValueChange` triggers `router.push` to the new `?type=` value — a full
   server round-trip, matching how `?status=` drill-down already works on `/purchase-orders` (Phase 5
   precedent, not a new pattern).

### Movements date-range filtering (D-07/D-08)

1. `page.tsx` reads `searchParams.from`/`searchParams.to` only when `type === "movements"`.
2. Each value is regex-validated (`/^\d{4}-\d{2}-\d{2}$/`) BEFORE constructing a `Date` — a non-matching or
   absent value falls back to the default 30-day range (`gte: thirtyDaysAgo`, `lte: now`), never throwing
   (RESEARCH.md Pattern 2 — this is a NEW guard, do not copy `/inventory`'s current unguarded
   `new Date(params.from)` forward, per Pitfall 3).
3. The Movements tab's date `<Input type="date">` elements read/write the URL via the same
   `router.push`-on-change pattern already established in `inventory-client.tsx`'s `updateFilter()`.

### Export (D-04/D-05/D-06)

1. Each report tab renders a single `<a href={...} download>` (styled as an outline Button via the `render`
   prop) whose `href` is `/api/reports/{type}` with the SAME searchParams currently applied to the visible
   report (e.g., Movements exports with the exact `?from=&to=` currently shown).
2. Clicking it is a plain browser navigation/download — no `fetch`, no blob URL construction, no loading
   spinner state (D-06).
3. The Route Handler independently re-derives and re-validates its own query from
   `request.nextUrl.searchParams` — it does NOT trust or reuse the page's already-rendered data (RESEARCH.md
   Pattern 3), guaranteeing D-04's "export matches exactly what's visible" contract even though there is no
   shared cache between the page request and the later `<a href>` navigation.

### Query-param validation (RESEARCH.md Pitfall 2/3, Security Domain V5 — hard requirement)

Both `?type=` and `?from=`/`?to=` MUST use whitelist/regex-then-fallback validation in BOTH the page AND each
Route Handler independently — never pass an unvalidated string into a Prisma `where` clause:
- `?type=`: only the exact literals `"inventory"`, `"movements"`, `"purchase-orders"` are accepted;
  anything else (wrong case, unrelated value, absent) → `"inventory"` default, no error.
- `?from=`/`?to=`: only `/^\d{4}-\d{2}-\d{2}$/`-shaped strings are accepted; anything else → 30-day default,
  no error. This is the SAME discipline already established for `?stock=`/`?status=` in Phase 5 — do not
  regress to `/inventory`'s current unguarded pattern (T-03-11).
