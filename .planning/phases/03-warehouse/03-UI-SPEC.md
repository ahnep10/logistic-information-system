> Visual and interaction contract for Phase 3: Warehouse (Stock Transactions + Inventory History).
> Generated by gsd-ui-researcher. Consumed by gsd-planner, gsd-executor, gsd-ui-checker, gsd-ui-auditor.

### Transaction Type Badge Colors (Phase 3 new)

Phase 3 introduces IN/OUT type indicator badges on both tables. They use the same soft-tint pattern as
severity badges to maintain visual consistency. Do NOT use the badge `destructive` variant for OUT — it has
a solid fill that is too harsh in data tables.

| Type | className | Label |
|------|-----------|-------|
| STOCK_IN | `bg-green-100 text-green-700 border border-green-200 hover:bg-green-100` | `IN` |
| STOCK_OUT | `bg-red-100 text-red-700 border border-red-200 hover:bg-red-100` | `OUT` |

**Implementation pattern:**
```tsx
function getTypeBadgeClass(type: "STOCK_IN" | "STOCK_OUT") {
  return type === "STOCK_IN"
    ? "bg-green-100 text-green-700 border border-green-200 hover:bg-green-100"
    : "bg-red-100 text-red-700 border border-red-200 hover:bg-red-100"
}

<Badge className={getTypeBadgeClass(tx.type)}>
  {tx.type === "STOCK_IN" ? "IN" : "OUT"}
</Badge>
```

**INVT-06 severity display on /inventory:** The /inventory page does NOT show a severity column per row.
Severity (Critical/Warning/OK) is a property of a product's current stock, not of an individual transaction.
INVT-06 is satisfied by the /products page (Phase 2) and the /stock Recent Transactions table does not need
it either. Executor: do NOT add a severity column to the inventory history table. The requirement is met
by the existing products page.

**Suspense boundary — mandatory:**
```tsx
// page.tsx (Server Component)
import { Suspense } from "react"
import InventoryClient from "./inventory-client"

export default async function InventoryPage({ searchParams }) {
  const params = await searchParams  // MUST await in Next.js 15
  const [transactions, products] = await Promise.all([...])
  return (
    <Suspense fallback={<div className="p-6 text-sm text-muted-foreground">Loading...</div>}>
      <InventoryClient transactions={transactions} products={products} currentParams={params} />
    </Suspense>
  )
}
```
`InventoryClient` uses `useSearchParams` — Next.js 15 requires it inside `<Suspense>`. Missing this causes
a build error. See RESEARCH.md Pitfall 2.

30 days, show empty state.

## Interaction Contracts

### Stock In / Stock Out form submission pattern

1. User opens dialog, fills form, clicks submit.
2. Client-side Zod validation (`stockInSchema` / `stockOutSchema` from `lib/validations/stock-transaction.ts`) —
   invalid fields show `FormMessage` inline below each field.
3. If client valid: submit button enters Loading state; Server Action (`recordStockIn` / `recordStockOut`)
   invoked.
4. Server Action: `auth()` session check → Zod `safeParse` → `prisma.$transaction()` atomic mutation.
5. **On server validation error:** return `{ error: "Invalid input. Please check all fields." }` — render as
   inline `<p className="text-sm text-destructive">` above the footer.
6. **On insufficient stock (stock-out only):** return `{ error: "Insufficient stock. Current stock: N units." }`
   — render as `<p className="text-sm text-destructive">` above the footer. Dialog stays open. Form NOT reset.
7. **On success:** `revalidatePath("/stock")` + `revalidatePath("/inventory")` + `revalidatePath("/products")`.
   Dialog closes programmatically. Recent Transactions table refreshes (RSC revalidation).

### Filter controls URL-push pattern (/inventory)

1. Any filter change (product select, date input blur, tab change) calls `updateFilter(key, value)` in
   `InventoryClient`.
2. `updateFilter` builds a new `URLSearchParams`, sets or deletes the key, then calls
   `router.push(`${pathname}?${newParams.toString()}`)`.
3. Next.js Server Component re-renders with the new `searchParams`, fetches filtered data from Prisma,
   passes updated `transactions` prop to `InventoryClient` via RSC re-render.
4. No client-side array filtering — all filtering is Prisma-level for correctness.

### Dialog open/close pattern (inherited from Phases 1–2)

- Dialog opens when user clicks CTA button.
- "Discard" closes the dialog without submitting — discards unsaved input.
- Escape key closes (base-ui Dialog built-in).
- Dialog closes programmatically on success.
- Form state resets on close (do not persist partial input across open/close cycles).
- `render` prop pattern: `<DialogTrigger render={<Button>Record Stock In</Button>}>`

### Date format contract

All timestamps displayed as: `"Jul 1, 2026, 14:32"` (no seconds, 24h or locale-appropriate).
Implementation:
```ts
new Date(tx.createdAt).toLocaleString("en-US", {
  month: "short", day: "numeric", year: "numeric",
  hour: "2-digit", minute: "2-digit"
})
```
This must be consistent across both the Recent Transactions table (/stock) and the Inventory History table (/inventory).

### Reason storage convention

Store as the display label string (e.g., `"Purchase Received"`, `"Write-Off"`). The Zod enum validates the
exact string. No kebab-slug transformation. The history table renders the stored value directly without
transformation.
