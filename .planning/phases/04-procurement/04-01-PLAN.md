## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Action → Prisma → Postgres | All PO mutations (04-03/04-04) cross this boundary; the schema built here is the last line of defense if application-level validation is bypassed |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Tampering | `PurchaseOrder.totalAmount` / `PurchaseOrderLineItem.unitPrice` precision | medium | mitigate | Store as `Decimal(12,2)`, never `Float` — prevents floating-point rounding drift from silently corrupting a persisted currency total across repeated Draft edits (D-23's own premise assumes exact Decimal arithmetic) |
| T-04-02 | Elevation of Privilege / Integrity | `PurchaseOrderLineItem` orphaning on Draft delete | medium | mitigate | `onDelete: Cascade` on the `PurchaseOrderLineItem.purchaseOrder` relation — DB-level, cannot be forgotten by a future code path (RESEARCH.md "Don't Hand-Roll") |
| T-04-03 | Repudiation | Missing single choke-point for the RECEIVED-immutability rule (D-17) | high | mitigate | `assertPOEditable(status)` is defined once here and must be the single function every mutating Server Action (04-04) calls before writing — centralizing this guard prevents one action from forgetting the check |

</threat_model>
