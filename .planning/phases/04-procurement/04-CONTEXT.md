# Phase 4: Procurement - Context

**Gathered:** 2026-07-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Build the purchase order lifecycle: Prisma `PurchaseOrder` + `PurchaseOrderLineItem` models, a dedicated `/purchase-orders/new` creation page, a `/purchase-orders` list page with status filtering, and a `/purchase-orders/[id]` detail page that drives the Draft ‚Üí Ordered ‚Üí Received transitions. Receiving a PO atomically creates one `StockTransaction` per line item and updates `Product.currentStock` in the same DB transaction as the status change to Received (reusing Phase 3's atomic stock-mutation pattern). Received POs are immutable. No partial-receipt state ‚Äî a PO is received in full in one action, though staff can correct the actual received quantity per line before confirming.

Stub page at `/purchase-orders` exists from Phase 1 scaffolding ‚Äî replace it with the full list implementation.

</domain>

<decisions>
## Implementation Decisions

### Prisma Schema

- **D-01:** Add `PurchaseOrder` model: `id` (cuid), `poNumber` (Int, `@default(autoincrement())`, human-friendly display as `PO-{poNumber padded}`), `supplierId` (FK ‚Üí Supplier), `status` (new `POStatus` enum: `DRAFT` | `ORDERED` | `RECEIVED`), `totalAmount` (Decimal ‚Äî sum of line items, recalculated server-side on save/confirm), `createdById` (FK ‚Üí User), `createdAt`, `updatedAt`.
- **D-02:** Add `PurchaseOrderLineItem` model: `id` (cuid), `purchaseOrderId` (FK ‚Üí PurchaseOrder), `productId` (FK ‚Üí Product), `quantity` (Int, ordered quantity), `unitPrice` (Decimal, manually entered ‚Äî no product cost field exists), `receivedQuantity` (Int?, nullable until received).
- **D-03:** Add nullable `purchaseOrderId` FK on the existing `StockTransaction` model (from Phase 3) so stock-in transactions generated by a PO receipt can be traced back to their originating PO. This is additive ‚Äî no change to Phase 3 behavior for manual stock-in/out.
- **D-04:** Add `purchaseOrders PurchaseOrder[]` relation to `Supplier` and `User` models; `purchaseOrderLineItems PurchaseOrderLineItem[]` to `Product`.

### PO Creation & Line Items

- **D-05:** Draft PO creation is a dedicated page (`/purchase-orders/new`), not a Dialog ‚Äî line items are a repeating, growable list that needs more room than the Dialog pattern used for Products/Suppliers/Categories.
- **D-06:** Line items are added one at a time via an inline row form (Product select ‚Äî `isActive: true` products only, per Phase 2/3 convention ‚Äî Quantity, Unit Price) with an "Add Line" button that appends to an editable/removable table below.
- **D-07:** Total (`totalAmount`) is auto-calculated client-side as `sum(quantity √ó unitPrice)` across all line items and shown live; recalculated server-side on every save as the source of truth (never trust client-submitted total).
- **D-08:** A Draft can be saved with zero line items (work-in-progress). Confirming Draft ‚Üí Ordered requires at least 1 line item ‚Äî Server Action rejects with a clear error otherwise.
- **D-09:** Unit price is always manually entered per line item. No product cost/price field is added to the `Product` schema ‚Äî that would be scope creep beyond Phase 2's model. If a "default cost" field becomes valuable later, it's a separate phase/decision.

### Goods Receipt Flow (PROC-03)

- **D-10:** No partial-receipt state. A PO is received in one action that advances it directly from Ordered to Received ‚Äî there is no "Partially Received" status and no remaining-balance tracking across multiple receipt events.
- **D-11:** The receive screen pre-fills each line item's received quantity with the ordered quantity, but staff can edit it before confirming (handles real-world discrepancies like short shipments). This is still a single all-lines-at-once confirmation, not a running partial state.
- **D-12:** Receiving creates exactly one `StockTransaction` per line item (matching the existing schema's one-product-per-row shape), each with `quantity = receivedQuantity` (the staff-corrected value, not the original ordered quantity). All line-item stock transactions plus the PO status update to `RECEIVED` happen inside a single `prisma.$transaction`, reusing the Phase 3 D-05/D-06/D-07 atomic-mutation pattern (including the no-negative-stock guard, though stock-in can never go negative).
- **D-13:** Generated stock-in transactions use `reason = "Purchase Received"` (reusing the exact Phase 3 D-02 label) and set the new `purchaseOrderId` FK to link back to the PO.
- **D-22:** The goods-receipt Server Action must row-lock the `PurchaseOrder` itself, not just the `Product` rows. Inside the same `prisma.$transaction`, issue `SELECT ... FOR UPDATE` on the target `PurchaseOrder` row (via `tx.$queryRaw`, matching the Phase 3 D-05/03-01 `SELECT FOR UPDATE` convention), then re-check `status === "ORDERED"` *after* the lock is acquired ‚Äî before creating any `StockTransaction` rows or updating status to `RECEIVED`. If the re-check fails (already received by a concurrent request), throw and return an error (e.g. `{ error: "This purchase order has already been received." }`). This closes a double-receipt race: without locking the PO row, two concurrent receive attempts (double-click, retry, two staff members) could both pass the status check before either commits, producing duplicate `StockTransaction` rows and double-incrementing `Product.currentStock` for one physical shipment ‚Äî violating the "Received is immutable, one atomic action" guarantee (D-10/D-17).

### Status Transitions & Permissions

- **D-14:** Both Manager and Staff can create, edit (Draft only), confirm (Draft ‚Üí Ordered), and receive (Ordered ‚Üí Received) POs. No `requireManager()`-style restriction ‚Äî same authenticated-session-only guard used for Phase 3 stock transactions. Manager is a permissions superset throughout the app, never more restricted than Staff.
- **D-15:** A Draft PO can be hard-deleted (not soft-deleted ‚Äî nothing has been committed to stock/history yet). Delete button only appears while `status === "DRAFT"`, behind a confirm dialog (reuse `alert-dialog.tsx`).
- **D-16:** Confirming Draft ‚Üí Ordered re-validates that the supplier and every line-item product are still `isActive: true` at confirm time. If any was deactivated after the Draft was started, confirmation is blocked with a clear error naming the deactivated entity.
- **D-17:** PROC-04 immutability: Server Actions for edit/confirm/receive/delete all reject if `status === "RECEIVED"` (the real security layer). The PO detail page additionally hides all action buttons and shows a "Received" badge when status is Received ‚Äî UI convenience, not the enforcement mechanism.

### PO List & Detail Views

- **D-18:** `/purchase-orders` list page uses Tabs for status filtering: All / Draft / Ordered / Received ‚Äî same client-side Tabs-over-server-fetched-data pattern as Phase 2's Suppliers active/inactive filter.
- **D-19:** List table columns: PO # (formatted `PO-0001`), Supplier, Status (Badge), Total, Created date, Created by.
- **D-20:** PO detail is a dedicated page at `/purchase-orders/[id]`, not a Dialog/Sheet ‚Äî shows header info (supplier, status, dates, created by), full line-item table with per-line subtotal and grand total, and contextual action buttons (Edit/Confirm/Receive/Delete) based on current status. Same dedicated-page rationale as D-05; also where the receive-quantity-editing UI (D-11) lives.
- **D-21:** `poNumber` is formatted for display as `PO-{poNumber, zero-padded to 4 digits}` (e.g., `PO-0001`) ‚Äî never the raw cuid `id`, which is impractical for staff to reference verbally or on paper during receiving.

### Decimal Serialization (Server ‚Üí Client boundary)

- **D-23:** `PurchaseOrder.totalAmount`, `PurchaseOrderLineItem.unitPrice`, and any computed line subtotals are Prisma `Decimal` instances (class objects with methods), not plain JS primitives. Every Decimal value MUST be explicitly converted via `.toNumber()` (for arithmetic/display formatting) or `.toString()` inside the Server Component ‚Äî before being passed as props to any Client Component. This applies to the `/purchase-orders` list page, the `/purchase-orders/[id]` detail page, and the receive-quantity-editing UI. Passing a raw Decimal instance across the Server‚ÜíClient boundary throws "Only plain objects can be passed to Client Components from Server Components" ‚Äî the same class of RSC serialization error already hit once in this codebase with Lucide icon components (Phase 1, `01-02-SUMMARY.md`: fixed by passing `ReactNode` instead of a component reference). Server Actions receiving these values back from client-submitted forms must parse the string/number back into `Decimal` (e.g. `new Prisma.Decimal(value)`) before writing to the DB.

### Claude's Discretion

- Exact Prisma `Decimal` vs `Int` (cents) internal storage representation for `unitPrice`/`totalAmount` ‚Äî Claude decides based on Prisma 6 Decimal handling and existing schema conventions (no money fields exist yet in this codebase). Regardless of storage choice, D-23's Server‚ÜíClient conversion rule applies to whatever the DB layer returns.
- Zod validation schema location: `lib/validations/purchase-order.ts`, following the established pattern.
- Line-item table styling and responsive behavior on the create/detail pages ‚Äî follow Phase 2/3 table conventions.
- Badge variants for DRAFT/ORDERED/RECEIVED status ‚Äî pick colors consistent with existing severity/type badge conventions (e.g., neutral/blue/green).
- Whether "Edit Draft" reuses the same `/purchase-orders/new`-style form or a dedicated `/purchase-orders/[id]/edit` route ‚Äî functionally equivalent, Claude picks the simpler implementation.
- `poNumber` padding width (4 digits assumed) ‚Äî adjust if a different width reads better.

</decisions>
