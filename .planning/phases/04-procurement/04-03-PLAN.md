<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser (form submission) → Server Action → Prisma | User-controlled `supplierId`/`lineItems` cross this boundary as `FormData` (including a JSON-encoded nested array — a genuinely new input shape for this codebase) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-06 | Tampering | Client-submitted `totalAmount` | high | mitigate | The create/edit form never sends a `totalAmount` field at all — the server always computes it from the persisted `lineItems` array via `Prisma.Decimal` arithmetic (D-07); there is no client value to tamper with |
| T-04-07 | Tampering | Draft PO referencing a deactivated supplier/product | medium | mitigate | `new/page.tsx` only fetches `isActive: true` suppliers/products for the dropdowns (matches Phase 2/3 convention); this is a first line of defense — the authoritative re-check at confirm time is D-16, implemented in 04-04 |
| T-04-08 | Tampering / Injection | Malformed or crafted JSON in the `lineItems` FormData field | medium | mitigate | `JSON.parse` wrapped in try/catch (parse failure → generic error, never thrown to the client), and the parsed result is always passed through `createPurchaseOrderSchema.safeParse` before any DB write — a structurally invalid or unexpected-shape payload is rejected by Zod, never used to build a raw query |

</threat_model>
