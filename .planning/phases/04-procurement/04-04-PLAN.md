Row-locked atomic transaction to extend (RESEARCH.md Pattern 3 / PATTERNS.md — the exact technique already proven in `actions/stock-transactions.ts`'s `SELECT ... FOR UPDATE` via `tx.$queryRaw`, applied here to the `PurchaseOrder` parent row instead of a `Product` row):
```
await prisma.$transaction(async (tx) => {
  const rows = await tx.$queryRaw<Array<{ status: string }>>`
    SELECT "status" FROM purchase_orders WHERE id = ${purchaseOrderId} FOR UPDATE
  `
  if (rows.length === 0) throw new Error("Purchase order not found.")
  if (rows[0].status !== "ORDERED") {
    throw new Error("This purchase order has already been received.")
  }
  for (const line of lineItems) {
    await tx.product.update({ where: { id: line.productId }, data: { currentStock: { increment: line.receivedQuantity } } })
    await tx.stockTransaction.create({ data: { type: "STOCK_IN", productId: line.productId, quantity: line.receivedQuantity, reason: "Purchase Received", purchaseOrderId, createdById: session.user.id } })
  }
  await tx.purchaseOrder.update({ where: { id: purchaseOrderId }, data: { status: "RECEIVED" } })
})
```
The row-lock query MUST be the first statement inside the transaction and the status check MUST happen strictly after it returns, before any other read or write (D-22 — this ordering is the entire mitigation for the double-receipt race).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Concurrent Server Action invocations → same `PurchaseOrder` row | Two overlapping `receivePurchaseOrder` calls (double-click, retry, two staff members) racing against one physical shipment |
| Server Action → direct invocation bypassing UI | Any of `confirmPurchaseOrder`/`receivePurchaseOrder`/`deletePurchaseOrder` can be called directly (e.g. via browser devtools replaying a request), independent of which buttons the UI currently renders |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-09 | Tampering / Repudiation | `receivePurchaseOrder` double-receipt race | high | mitigate | `SELECT ... FOR UPDATE` row lock on the `PurchaseOrder` row is the first statement inside `prisma.$transaction`, and the `status !== "ORDERED"` check happens strictly after that query returns, before any `StockTransaction` create or `Product.currentStock` update (D-22) — closes the race where two concurrent requests both read `status === "ORDERED"` before either commits |
| T-04-10 | Tampering | Confirming a Draft against a since-deactivated supplier/product | medium | mitigate | `confirmPurchaseOrder` re-queries `supplier.isActive` and every line item's `product.isActive` at confirm time, not the state captured when the Draft was created (D-16) |
| T-04-11 | Elevation of Privilege / Tampering | Post-receipt mutation attempt (edit/confirm/receive/delete on a `RECEIVED` PO) bypassing hidden UI buttons | high | mitigate | Every mutating action (`confirmPurchaseOrder`, `deletePurchaseOrder` via `assertPOEditable`; `receivePurchaseOrder` via its own status re-check under the row lock) rejects server-side regardless of UI state — button-hiding in `po-detail-client.tsx` is cosmetic only (D-17) |
| T-04-12 | Tampering | SQL injection via the `tx.$queryRaw` template literal in `receivePurchaseOrder` | high | mitigate | Prisma's tagged-template `$queryRaw` automatically parameterizes the interpolated `${purchaseOrderId}` — the exact same safe pattern already in production in `actions/stock-transactions.ts`; never switch to `$queryRawUnsafe` or string concatenation |
| T-04-13 | Tampering | Hard-deleting a non-Draft PO via a forged direct call | medium | mitigate | `deletePurchaseOrder` re-fetches the current `status` server-side and rejects unless exactly `"DRAFT"`, regardless of which buttons the requesting client happened to render (D-15) |

</threat_model>
