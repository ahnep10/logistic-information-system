### PO Status Badge Colors (Phase 4 new — Claude's discretion per CONTEXT.md, resolved here)

Same soft-tint pattern as severity/transaction-type badges established in Phases 2–3 (light background +
dark text + border). Do NOT use the `destructive` Badge variant for any status — it is reserved for the
Delete confirmation only.

| Status | className | Label |
|--------|-----------|-------|
| DRAFT | `bg-slate-100 text-slate-700 border border-slate-200 hover:bg-slate-100` | `Draft` |
| ORDERED | `bg-blue-100 text-blue-700 border border-blue-200 hover:bg-blue-100` | `Ordered` |
| RECEIVED | `bg-green-100 text-green-700 border border-green-200 hover:bg-green-100` | `Received` |

**Implementation pattern — new file `lib/utils/po-status.ts` (mirrors `lib/utils/severity.ts` convention):**
```typescript
export type POStatus = "DRAFT" | "ORDERED" | "RECEIVED"

export function getStatusBadge(status: POStatus): { label: string; className: string } {
  switch (status) {
    case "DRAFT":
      return { label: "Draft", className: "bg-slate-100 text-slate-700 border border-slate-200 hover:bg-slate-100" }
    case "ORDERED":
      return { label: "Ordered", className: "bg-blue-100 text-blue-700 border border-blue-200 hover:bg-blue-100" }
    case "RECEIVED":
      return { label: "Received", className: "bg-green-100 text-green-700 border border-green-200 hover:bg-green-100" }
  }
}
```
Import `getStatusBadge` everywhere a status badge is rendered (`/purchase-orders` list table, PO detail
page header). Do not duplicate this logic inline.

### Currency Display (Phase 4 new — default, not locked by CONTEXT.md)

No prior phase introduced a money field, and CONTEXT.md/RESEARCH.md do not lock a currency-formatting
convention. Given the project's Indonesian-market context (existing supplier phone placeholder
`+62 21 1234 5678` from 02-UI-SPEC.md), the default is Indonesian Rupiah formatting with no decimal places
(matches whole-currency-unit conventions common in ID business software):

```typescript
new Intl.NumberFormat("id-ID", {
  style: "currency",
  currency: "IDR",
  minimumFractionDigits: 0,
}).format(amount) // e.g. "Rp1.250.000"
```

Applies to: line-item Unit Price display, line-item Subtotal, and the grand Total on all three PO screens.
Input fields (Unit Price entry) remain plain `type="number"` — currency formatting is display-only, not
applied to the editable input value. **This is a default, flag during checker review if the project intends
a different currency/locale.**

### Screen 2: Create Purchase Order Page (`/purchase-orders/new`)

**Access:** Both MANAGER and STAFF (D-14). Dedicated page, not a Dialog (D-05) — line items are a
repeating, growable list needing more room than the Dialog pattern used for Products/Suppliers/Categories.

**Focal point:** The line-item table and the "Add Line" affordance — this is a data-entry-heavy page, not a
single-CTA page. The live Total is the secondary focal point (bottom-right, above the submit action).

**Layout:**
```
[Page content — max-w-4xl mx-auto p-6]
├── Page header [flex items-center justify-between mb-6]
│     ├── "Create Purchase Order" [text-2xl font-semibold]
│     └── Button "Cancel" [variant="ghost"] — Link back to /purchase-orders
├── Card "Supplier" [mb-6]
│     └── CardContent
│           └── Supplier Select [label: "Supplier", required]
│                 — only isActive:true suppliers (D-06/D-16 convention)
│                 — placeholder: "Select a supplier"
├── Card "Line Items" [mb-6]
│     └── CardContent [p-0]
│           ├── Inline add-line row [flex gap-3 p-4 border-b] (D-06)
│           │     ├── Product Select [flex-1, isActive:true products only]
│           │     │     — placeholder: "Select a product"
│           │     ├── Quantity input [w-24, type: number, min: 1, placeholder: "Qty"]
│           │     ├── Unit Price input [w-36, type: number, min: 0, step: 0.01, placeholder: "Unit price"]
│           │     └── Button "Add Line" [variant="outline"] — Plus icon [h-4 w-4 mr-2]
│           │           — appends to fields array via useFieldArray.append()
│           └── Table (added line items, empty if none added yet)
│                 ├── TableHeader: Product | Quantity | Unit Price | Subtotal | (remove icon col, 48px)
│                 └── TableBody — one row per field.id (RHF field.id as React key, NOT array index)
│                       └── Each row: read-only product name + qty + unit price + computed subtotal
│                             + Button [variant="ghost" size="icon"] Trash2 icon → fields.remove(index)
├── Total row [flex justify-end items-center gap-3 py-4]
│     ├── "Total" [text-sm font-semibold text-muted-foreground]
│     └── Live-computed sum(qty × unitPrice) [text-xl font-semibold] — client preview only (D-07)
└── Footer actions [flex justify-end gap-3]
      ├── Button "Cancel" [variant="outline"] — Link to /purchase-orders, discards unsaved input
      └── Button "Save Draft" [bg-primary, shows Loader2 spinner on submit]
            — allowed with 0 line items (D-08) — no client-side minimum-line validation on this page
```

**Add-line row behavior:** "Add Line" is disabled until Product, Quantity (>0), and Unit Price (>=0) are
all filled in the inline row. On click: `append({ productId, quantity, unitPrice })` via `useFieldArray`,
then the inline row resets to empty for the next entry. This is a controlled convenience row, separate from
the `fields` array itself (per RESEARCH.md Pattern 4).

**On successful save:** Server Action `createDraftPurchaseOrder` recomputes `totalAmount` server-side
(D-07), assigns `poNumber` via DB sequence, and redirects to `/purchase-orders/[id]` (the newly created
Draft's detail page) — not back to the list. This lets the user immediately continue toward "Confirm Order"
or add more lines from the detail page's Draft-state edit view (Screen 3).

**Empty line-items table (before any line added):**
```
  [flex flex-col items-center py-8 text-center]
  ├── "No line items added yet" [text-sm font-medium]
  └── "Use the fields above to add a product line." [text-sm text-muted-foreground mt-1]
```

### Screen 3: Purchase Order Detail Page (`/purchase-orders/[id]`)

**Access:** Both MANAGER and STAFF (D-14). Dedicated page (D-20). Layout and available actions are
**status-conditional** — three distinct states below. The page header (PO #, Supplier, Status badge,
Created date, Created by) is constant across all three states; only the body content and action buttons
change.

**Shared header (all states):**
```
[Page content — max-w-4xl mx-auto p-6]
├── Page header [flex items-center justify-between mb-6]
│     ├── Left: formatPONumber(poNumber) [text-2xl font-semibold] + Badge (getStatusBadge) [ml-3]
│     └── Right: status-conditional action buttons (see below)
├── Card "Details" [mb-6]
│     └── CardContent [grid grid-cols-2 gap-4]
│           ├── Supplier: name [text-sm], label "Supplier" [text-xs font-semibold text-muted-foreground]
│           ├── Created: formatted date + createdBy.name, label "Created"
│           └── (Draft state only: Supplier is an editable Select, not static text — see below)
```

**State A — DRAFT (editable):**
```
├── Action buttons (header right): 
│     ├── Button "Delete Draft" [variant="ghost", text-destructive] — Trash2 icon — opens AlertDialog (D-15)
│     └── Button "Confirm Order" [bg-primary] — opens AlertDialog before executing (irreversible edit-lock)
├── Card "Details" — Supplier field IS an editable Select (isActive:true suppliers only)
├── Card "Line Items" — SAME editable add-line-row + removable-table UI as Screen 2 (shared component)
├── Total row — live client preview, same as Screen 2
└── Footer: Button "Save Draft" [bg-primary] — persists edits via updateDraftPurchaseOrder Server Action
```
Reuses the exact same line-item form component as `/purchase-orders/new` (Claude's discretion: no separate
`/purchase-orders/[id]/edit` route — the detail page IS the edit page while status is Draft). This
satisfies "Whether Edit Draft reuses the same new-style form" discretion item in CONTEXT.md: yes, reused.

**Confirm Order AlertDialog:**
```
AlertDialog
├── AlertDialogHeader
│     ├── AlertDialogTitle: "Confirm this purchase order?"
│     └── AlertDialogDescription:
│           "Once confirmed, line items can no longer be edited. You can still adjust
│            received quantities later when the goods arrive."
└── AlertDialogFooter
      ├── AlertDialogCancel: "Keep editing"
      └── AlertDialogAction [bg-primary]: "Confirm Order"
```
On confirm: Server Action `confirmPurchaseOrder` re-validates supplier + all line-item products are still
`isActive: true` (D-16) and that `lineItems.length >= 1` (D-08). On success: status → ORDERED, page
re-renders in State B. On failure: AlertDialog closes, inline error banner appears above the Details card
(see Error States below) — the dialog does not stay open with an error inside it (AlertDialog is a
one-shot confirm, not a form).

**Delete Draft AlertDialog (D-15):**
```
AlertDialog
├── AlertDialogHeader
│     ├── AlertDialogTitle: "Delete this purchase order?"
│     └── AlertDialogDescription:
│           "This draft has not been ordered yet. Deleting it is permanent and cannot be undone."
└── AlertDialogFooter
      ├── AlertDialogCancel: "Cancel"
      └── AlertDialogAction [variant="destructive"]: "Delete Draft"
```
On confirm: `deletePurchaseOrder` Server Action hard-deletes (cascades to line items), redirects to
`/purchase-orders`.

**State B — ORDERED (read-only, receive available):**
```
├── Action buttons (header right):
│     └── Button "Receive Goods" [bg-primary] — PackageCheck icon — toggles page into receive-edit mode
├── Card "Details" — Supplier is static text (read-only, D-17: no edits possible post-confirm)
├── Card "Line Items" — READ-ONLY table: Product | Quantity | Unit Price | Subtotal (no remove column)
└── Total row — server-computed totalAmount (source of truth), static, right-aligned
```

**Receive-edit mode (triggered by "Receive Goods" click, same page, no navigation — D-11/D-20):**
```
├── Card "Line Items" — table becomes: Product | Ordered Qty | Received Qty (editable input) | Notes(n/a)
│     — each Received Qty input pre-filled with the ordered quantity (D-11)
│     — input type="number", min: 0 (per RESEARCH.md Open Question 1 recommendation — 0 is valid, meaning
│       "this line was not delivered", not a partial-receipt state machine)
├── [Receive error banner — conditional, see Error States]
└── Footer actions [flex justify-end gap-3]
      ├── Button "Cancel" [variant="outline"] — exits receive-edit mode, discards quantity edits
      └── Button "Confirm Receipt" [bg-primary, shows Loader2 spinner on submit]
```
No separate AlertDialog before "Confirm Receipt" — the receive-edit table itself IS the explicit review
step (staff sees and can correct every quantity before submitting). Adding a second confirm dialog on top
would be redundant friction for what is already a deliberate, visible-state action.

On submit: Server Action `receivePurchaseOrder` executes the atomic transaction (D-22/D-12: row-lock PO,
re-check status, create StockTransactions, increment Product.currentStock, update status to RECEIVED).
On success: page re-renders in State C. On failure (double-receipt race, D-22): inline error banner, stays
in receive-edit mode with entered quantities preserved (form NOT reset — same convention as Phase 3's
insufficient-stock error).

**State C — RECEIVED (fully immutable):**
```
├── Action buttons (header right): none — no buttons rendered at all (D-17, UI convenience only)
├── Card "Details" — all fields static text
├── Card "Line Items" — READ-ONLY table: Product | Ordered Qty | Received Qty | Unit Price | Subtotal
└── Total row — server-computed totalAmount, static
```
The "Received" badge next to the PO number (shared header) is the only visual indicator needed — no
additional banner or lock icon required. D-17: server-side `status === "RECEIVED"` rejection in every
mutation Server Action is the actual enforcement; hiding buttons here is convenience only.

## Interaction Contracts

### Draft creation/edit submission pattern

1. User fills Supplier select + adds 0-or-more line items via the inline add-line row (`useFieldArray`).
2. Client-side Zod validation on the inline row before "Add Line" is enabled; no page-level submit
   validation blocks saving a Draft with 0 lines (D-08).
3. "Save Draft" invokes `createDraftPurchaseOrder` (new) or `updateDraftPurchaseOrder` (existing Draft
   being edited) — session check → Zod `safeParse` → server recomputes `totalAmount` via `Prisma.Decimal`
   arithmetic (D-07, never trust client total) → `prisma.purchaseOrder.create/update`.
4. On success: `revalidatePath("/purchase-orders")` + redirect to `/purchase-orders/[id]`.
5. On server validation error: inline `<p className="text-sm text-destructive">` above the footer,
   "Invalid input. Please check all fields."

### Confirm Order pattern (Draft → Ordered)

1. User clicks "Confirm Order" → AlertDialog opens (confirmation copy above).
2. On AlertDialog confirm: `confirmPurchaseOrder(id)` Server Action runs — session check → re-validate
   `supplier.isActive` and every line-item `product.isActive` (D-16) → reject if `lineItems.length === 0`
   (D-08) → update `status: "ORDERED"`.
3. On success: `revalidatePath("/purchase-orders")` + `revalidatePath("/purchase-orders/[id]")`; page
   re-renders in State B (Ordered, read-only + Receive Goods button).
4. On failure: AlertDialog closes; page re-renders with an inline error banner above the Details card
   naming the specific blocking condition (zero lines, or the specific deactivated entity per D-16).

### Goods Receipt pattern (Ordered → Received) — the critical atomic path

1. User clicks "Receive Goods" → page toggles into receive-edit mode (no navigation, no dialog) — each
   line's Received Qty input is pre-filled with its ordered quantity (D-11).
2. User optionally edits any Received Qty value (handles short shipments/discrepancies), then clicks
   "Confirm Receipt".
3. `receivePurchaseOrder(id, lineItemQuantities)` Server Action runs inside `prisma.$transaction`:
   `SELECT ... FOR UPDATE` on the PurchaseOrder row (via `tx.$queryRaw`) → re-check `status === "ORDERED"`
   strictly after lock acquisition (D-22) → per line item: `tx.product.update` increment +
   `tx.stockTransaction.create` with `reason: "Purchase Received"` and `purchaseOrderId` set (D-12/D-13) →
   `tx.purchaseOrder.update({ status: "RECEIVED" })`.
4. On success: `revalidatePath("/purchase-orders")`, `revalidatePath("/purchase-orders/[id]")`,
   `revalidatePath("/inventory")`, `revalidatePath("/stock")`, `revalidatePath("/products")`. Page
   re-renders in State C (Received, fully read-only).
5. On failure (already received — double-receipt race): inline error banner "This purchase order has
   already been received." — receive-edit mode stays open with entered quantities preserved (form NOT
   reset), mirroring the Phase 3 insufficient-stock error convention.

### Draft delete pattern (hard delete, D-15)

1. User clicks "Delete Draft" (only rendered when `status === "DRAFT"`) → AlertDialog opens.
2. On confirm: `deletePurchaseOrder(id)` Server Action — session check → reject if `status !== "DRAFT"` →
   `prisma.purchaseOrder.delete()` (cascades to line items via `onDelete: Cascade`).
3. On success: redirect to `/purchase-orders`.

### Decimal serialization boundary (D-23 — restated for executor convenience)

Every Prisma `Decimal` field (`totalAmount`, `unitPrice`, computed subtotals) MUST be converted via
`.toNumber()` inside the Server Component before being passed as a prop to any Client Component, on all
three screens. Server Actions receiving these values back from client form submissions must reconstruct
`new Prisma.Decimal(value)` before writing. See CONTEXT.md D-23 and RESEARCH.md Pattern 2/Pitfall 1 for the
full rationale — this UI-SPEC does not duplicate the code example, only flags every screen it applies to.

### Status filter pattern (`/purchase-orders` list)

Client-side Tabs filter over server-fetched data — identical mechanism to Phase 2's Suppliers
active/inactive filter and Phase 3's inventory type filter's sibling pattern: all POs fetched once
server-side (with `supplier`, `createdBy` included, `Decimal` fields pre-converted), Tabs selection filters
the already-fetched array in the client component. No URL param, no server round-trip on tab change
(matches D-18's explicit "same client-side Tabs pattern as Phase 2 Suppliers").

### Date format contract

All dates displayed as: `"Jul 3, 2026"` (date only, no time — POs are not high-frequency events like stock
transactions). Implementation:
```ts
new Date(po.createdAt).toLocaleDateString("en-US", { month: "short", day: "numeric", year: "numeric" })
```
Consistent across the list page's "Created" column and the detail page's "Created" field.
