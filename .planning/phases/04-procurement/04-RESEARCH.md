## User Constraints (from CONTEXT.md)

### Locked Decisions

**Prisma Schema**

- **D-01:** Add `PurchaseOrder` model: `id` (cuid), `poNumber` (Int, `@default(autoincrement())`, human-friendly display as `PO-{poNumber padded}`), `supplierId` (FK → Supplier), `status` (new `POStatus` enum: `DRAFT` | `ORDERED` | `RECEIVED`), `totalAmount` (Decimal — sum of line items, recalculated server-side on save/confirm), `createdById` (FK → User), `createdAt`, `updatedAt`.
- **D-02:** Add `PurchaseOrderLineItem` model: `id` (cuid), `purchaseOrderId` (FK → PurchaseOrder), `productId` (FK → Product), `quantity` (Int, ordered quantity), `unitPrice` (Decimal, manually entered — no product cost field exists), `receivedQuantity` (Int?, nullable until received).
- **D-03:** Add nullable `purchaseOrderId` FK on the existing `StockTransaction` model (from Phase 3) so stock-in transactions generated by a PO receipt can be traced back to their originating PO. This is additive — no change to Phase 3 behavior for manual stock-in/out.
- **D-04:** Add `purchaseOrders PurchaseOrder[]` relation to `Supplier` and `User` models; `purchaseOrderLineItems PurchaseOrderLineItem[]` to `Product`.

**PO Creation & Line Items**

- **D-05:** Draft PO creation is a dedicated page (`/purchase-orders/new`), not a Dialog — line items are a repeating, growable list that needs more room than the Dialog pattern used for Products/Suppliers/Categories.
- **D-06:** Line items are added one at a time via an inline row form (Product select — `isActive: true` products only, per Phase 2/3 convention — Quantity, Unit Price) with an "Add Line" button that appends to an editable/removable table below.
- **D-07:** Total (`totalAmount`) is auto-calculated client-side as `sum(quantity × unitPrice)` across all line items and shown live; recalculated server-side on every save as the source of truth (never trust client-submitted total).
- **D-08:** A Draft can be saved with zero line items (work-in-progress). Confirming Draft → Ordered requires at least 1 line item — Server Action rejects with a clear error otherwise.
- **D-09:** Unit price is always manually entered per line item. No product cost/price field is added to the `Product` schema — that would be scope creep beyond Phase 2's model. If a "default cost" field becomes valuable later, it's a separate phase/decision.

**Goods Receipt Flow (PROC-03)**

- **D-10:** No partial-receipt state. A PO is received in one action that advances it directly from Ordered to Received — there is no "Partially Received" status and no remaining-balance tracking across multiple receipt events.
- **D-11:** The receive screen pre-fills each line item's received quantity with the ordered quantity, but staff can edit it before confirming (handles real-world discrepancies like short shipments). This is still a single all-lines-at-once confirmation, not a running partial state.
- **D-12:** Receiving creates exactly one `StockTransaction` per line item (matching the existing schema's one-product-per-row shape), each with `quantity = receivedQuantity` (the staff-corrected value, not the original ordered quantity). All line-item stock transactions plus the PO status update to `RECEIVED` happen inside a single `prisma.$transaction`, reusing the Phase 3 D-05/D-06/D-07 atomic-mutation pattern (including the no-negative-stock guard, though stock-in can never go negative).
- **D-13:** Generated stock-in transactions use `reason = "Purchase Received"` (reusing the exact Phase 3 D-02 label) and set the new `purchaseOrderId` FK to link back to the PO.
- **D-22:** The goods-receipt Server Action must row-lock the `PurchaseOrder` itself, not just the `Product` rows. Inside the same `prisma.$transaction`, issue `SELECT ... FOR UPDATE` on the target `PurchaseOrder` row (via `tx.$queryRaw`, matching the Phase 3 D-05/03-01 `SELECT FOR UPDATE` convention), then re-check `status === "ORDERED"` *after* the lock is acquired — before creating any `StockTransaction` rows or updating status to `RECEIVED`. If the re-check fails (already received by a concurrent request), throw and return an error (e.g. `{ error: "This purchase order has already been received." }`). This closes a double-receipt race.

**Status Transitions & Permissions**

- **D-14:** Both Manager and Staff can create, edit (Draft only), confirm (Draft → Ordered), and receive (Ordered → Received) POs. No `requireManager()`-style restriction — same authenticated-session-only guard used for Phase 3 stock transactions. Manager is a permissions superset throughout the app, never more restricted than Staff.
- **D-15:** A Draft PO can be hard-deleted (not soft-deleted — nothing has been committed to stock/history yet). Delete button only appears while `status === "DRAFT"`, behind a confirm dialog (reuse `alert-dialog.tsx`).
- **D-16:** Confirming Draft → Ordered re-validates that the supplier and every line-item product are still `isActive: true` at confirm time. If any was deactivated after the Draft was started, confirmation is blocked with a clear error naming the deactivated entity.
- **D-17:** PROC-04 immutability: Server Actions for edit/confirm/receive/delete all reject if `status === "RECEIVED"` (the real security layer). The PO detail page additionally hides all action buttons and shows a "Received" badge when status is Received — UI convenience, not the enforcement mechanism.

**PO List & Detail Views**

- **D-18:** `/purchase-orders` list page uses Tabs for status filtering: All / Draft / Ordered / Received — same client-side Tabs-over-server-fetched-data pattern as Phase 2's Suppliers active/inactive filter.
- **D-19:** List table columns: PO # (formatted `PO-0001`), Supplier, Status (Badge), Total, Created date, Created by.
- **D-20:** PO detail is a dedicated page at `/purchase-orders/[id]`, not a Dialog/Sheet — shows header info (supplier, status, dates, created by), full line-item table with per-line subtotal and grand total, and contextual action buttons (Edit/Confirm/Receive/Delete) based on current status. Same dedicated-page rationale as D-05; also where the receive-quantity-editing UI (D-11) lives.
- **D-21:** `poNumber` is formatted for display as `PO-{poNumber, zero-padded to 4 digits}` (e.g., `PO-0001`) — never the raw cuid `id`, which is impractical for staff to reference verbally or on paper during receiving.

**Decimal Serialization (Server → Client boundary)**

- **D-23:** `PurchaseOrder.totalAmount`, `PurchaseOrderLineItem.unitPrice`, and any computed line subtotals are Prisma `Decimal` instances (class objects with methods), not plain JS primitives. Every Decimal value MUST be explicitly converted via `.toNumber()` (for arithmetic/display formatting) or `.toString()` inside the Server Component — before being passed as props to any Client Component. This applies to the `/purchase-orders` list page, the `/purchase-orders/[id]` detail page, and the receive-quantity-editing UI. Passing a raw Decimal instance across the Server→Client boundary throws "Only plain objects can be passed to Client Components from Server Components" — the same class of RSC serialization error already hit once in this codebase with Lucide icon components (Phase 1, `01-02-SUMMARY.md`: fixed by passing `ReactNode` instead of a component reference). Server Actions receiving these values back from client-submitted forms must parse the string/number back into `Decimal` (e.g. `new Prisma.Decimal(value)`) before writing to the DB.

## Summary

Phase 4 adds the purchase-order lifecycle on top of an already-established codebase with three prior phases of consistent conventions (Server Component + `xxx-client.tsx` split, `"use server"` Server Actions with Zod `safeParse`, `prisma.$transaction` with `SELECT ... FOR UPDATE` for atomic stock mutation). Every canonical reference file named in CONTEXT.md (`actions/stock-transactions.ts`, `actions/products.ts`, `actions/suppliers.ts`, `lib/validations/stock-transaction.ts`, `app/(protected)/inventory/page.tsx`, `prisma/schema.prisma`, plus `03-01-SUMMARY.md` and `01-02-SUMMARY.md`) was read directly and confirmed to exist with the exact shape CONTEXT.md assumes — no drift detected. The `/purchase-orders` stub page is a one-line placeholder as described.

The two genuinely new technical risks in this phase are (1) correctly serializing Prisma `Decimal` values across the Server→Client RSC boundary and back (D-23), and (2) extending the existing single-row `SELECT FOR UPDATE` pattern to a multi-row transaction that locks the `PurchaseOrder` parent row while writing multiple `StockTransaction` children plus a `Product.currentStock` increment per line item. Both are well-documented, low-risk patterns once the correct API (`.toNumber()`/`.toString()`, `new Prisma.Decimal(...)`, `tx.$queryRaw` before any writes) is followed — this is not novel engineering, it is disciplined repetition of Phase 3's proven pattern across a one-to-many relationship instead of a single row.

**Primary recommendation:** Reuse `actions/stock-transactions.ts` as the literal template for the goods-receipt action — same `prisma.$transaction(async (tx) => {...})` shape, extended with an additional `SELECT ... FOR UPDATE` on the `PurchaseOrder` row before the existing per-product lock/increment loop, and use `useFieldArray` (not manual `useState` array juggling) for the repeating line-item form to stay idiomatic with the installed `react-hook-form@7.80.0`.

        ▼

### Pattern 1: Prisma `Decimal` — server-side arithmetic, never native operators

**What:** `Decimal` fields are class instances (backed by `decimal.js`), not plain JS numbers. Native `*`/`+` either throws (TypeScript) or silently coerces incorrectly.
**When to use:** Any time `unitPrice` or `totalAmount` is read from Prisma and needs computation.
**Example:**
```typescript
// Source: prisma.io/docs/orm/prisma-client/special-fields-and-types [CITED]
import { Prisma } from "@prisma/client"

// Server Action — recompute totalAmount from line items (D-07, never trust client total)
const totalAmount = lineItems.reduce(
  (sum, li) => sum.plus(new Prisma.Decimal(li.quantity).times(li.unitPrice)),
  new Prisma.Decimal(0)
)

await prisma.purchaseOrder.update({
  where: { id },
  data: { totalAmount }, // Prisma accepts Decimal instances directly on write
})
```

### Pattern 2: Server→Client Decimal serialization (D-23)

**What:** Convert every `Decimal` field to a plain `number` or `string` inside the Server Component, before passing as a prop to any Client Component.
**When to use:** `/purchase-orders` list page, `/purchase-orders/[id]` detail page, receive-quantity UI.
**Example:**
```typescript
// Source: github.com/prisma/prisma/discussions/19983 [CITED — Prisma team guidance]
// app/(protected)/purchase-orders/page.tsx (Server Component)
const purchaseOrders = await prisma.purchaseOrder.findMany({
  include: { supplier: true, createdBy: true, lineItems: true },
})

const serialized = purchaseOrders.map((po) => ({
  ...po,
  totalAmount: po.totalAmount.toNumber(), // plain number — safe to pass to Client Component
  lineItems: po.lineItems.map((li) => ({
    ...li,
    unitPrice: li.unitPrice.toNumber(),
  })),
}))

return <PurchaseOrdersClient purchaseOrders={serialized} />
```
Server Actions receiving these back from client form submissions must reconstruct `Decimal`:
```typescript
const unitPrice = new Prisma.Decimal(formData.get("unitPrice") as string)
```

### Pattern 3: Row-locked multi-write transaction (extends Phase 3's single-row lock to PO + children)

**What:** Lock the `PurchaseOrder` row first, re-check status after lock acquisition, then perform all child writes (multiple `StockTransaction` creates + multiple `Product.currentStock` increments) inside the same transaction.
**When to use:** `receivePurchaseOrder` Server Action only (the one operation with a real concurrency risk — double-receipt).
**Example:**
```typescript
// Source: actions/stock-transactions.ts (existing codebase pattern, confirmed 03-01-SUMMARY.md)
await prisma.$transaction(async (tx) => {
  const rows = await tx.$queryRaw<Array<{ status: string }>>`
    SELECT "status" FROM purchase_orders WHERE id = ${purchaseOrderId} FOR UPDATE
  `
  if (rows.length === 0) throw new Error("Purchase order not found.")
  if (rows[0].status !== "ORDERED") {
    throw new Error("This purchase order has already been received.")
  }

  for (const line of lineItems) {
    await tx.product.update({
      where: { id: line.productId },
      data: { currentStock: { increment: line.receivedQuantity } },
    })
    await tx.stockTransaction.create({
      data: {
        type: "STOCK_IN",
        productId: line.productId,
        quantity: line.receivedQuantity,
        reason: "Purchase Received",
        purchaseOrderId,
        createdById: session.user.id,
      },
    })
  }

  await tx.purchaseOrder.update({
    where: { id: purchaseOrderId },
    data: { status: "RECEIVED" },
  })
})
```
**Note:** Unlike Phase 3's stock-out path, goods receipt is always a stock *increase* — the DB-level non-negative CHECK constraint on `products.currentStock` can never fire here, but the row lock is still required to prevent duplicate `StockTransaction` rows / double-increment from concurrent receive attempts (D-22's actual concern, not a negative-stock concern).

### Pattern 4: `useFieldArray` for the repeating line-item form

**What:** RHF's dedicated hook for array-of-objects form state.
**When to use:** PO creation/edit line-item table (D-06) and receive-quantity editing UI (D-11).
**Example:**
```typescript
// Source: react-hook-form.com/api/usefieldarray [CITED]
import { useFieldArray, useForm } from "react-hook-form"

const form = useForm<CreatePurchaseOrderInput>({
  resolver: zodResolver(createPurchaseOrderSchema) as any, // matches existing codebase cast convention
  defaultValues: { supplierId: "", lineItems: [] },
})

const { fields, append, remove } = useFieldArray({
  control: form.control,
  name: "lineItems",
})

// Add Line button (D-06)
<Button type="button" onClick={() => append({ productId: "", quantity: 1, unitPrice: 0 })}>
  Add Line
</Button>

// row render — use field.id (RHF-generated), NOT array index, as the React key
{fields.map((field, index) => (
  <TableRow key={field.id}>
    {/* FormField name={`lineItems.${index}.productId`} etc. */}
    <Button type="button" onClick={() => remove(index)}>Remove</Button>
  </TableRow>
))}
```

### Anti-Patterns to Avoid

- **Doing PO total arithmetic with native JS `*`/`+` on Decimal instances:** `Decimal * Decimal` either throws a TypeScript error or (if cast to `any`) produces `NaN`/string-concat bugs. Always use `.times()`/`.plus()` or convert to `number` first if precision loss is acceptable (it usually isn't for a persisted `totalAmount`).
- **Using array index as the React key in the line-item table:** breaks when a middle row is removed (RHF's `field.id` is stable across `remove()`/`append()` calls; index is not).
- **Checking `status === "ORDERED"` before acquiring the row lock:** this is exactly the race D-22 exists to prevent — the check must happen *after* `SELECT ... FOR UPDATE` returns, inside the transaction.
- **Trusting the client-submitted `totalAmount`:** always recompute server-side from the persisted line items (D-07).

|---------|-------------|-------------|-----|
| Repeating/growable line-item rows | Manual `useState<LineItem[]>` + custom add/remove/index-tracking logic | `react-hook-form`'s `useFieldArray` | Already installed, purpose-built, handles stable keys and validation wiring; manual re-implementation is strictly more code with more edge cases (e.g., re-indexing on remove) |
| PO sequential numbering | App-level counter table + manual increment-with-lock logic | Prisma `@default(autoincrement())` → Postgres `SEQUENCE` | Postgres sequences are already atomic and gap-tolerant by design (see Pitfall 2); a hand-rolled counter reintroduces exactly the race condition Postgres solves natively |
| Decimal-safe currency arithmetic | Custom fixed-point math on `number` | `Prisma.Decimal` (`decimal.js`) methods | Prisma already ships this; hand-rolled fixed-point math is a classic source of off-by-one-cent bugs |
| Cascade delete of line items when a Draft PO is deleted | Manual `deleteMany` on `PurchaseOrderLineItem` before deleting the parent | `onDelete: Cascade` on the `PurchaseOrderLineItem.purchaseOrder` relation | DB-level cascade is atomic and can't be forgotten in a future code path; app-level manual delete is a maintenance liability |

**Key insight:** Every "new" problem in this phase (sequencing, currency math, cascading deletes, repeating form rows) has a first-class Prisma/RHF primitive already installed in this project. The engineering discipline required is *not building around them* — Phase 3's `SELECT FOR UPDATE` precedent shows the team already defaults to the DB-native primitive over an app-level substitute, and this phase should hold that line.

## Common Pitfalls

### Pitfall 1: RSC "Only plain objects can be passed to Client Components" thrown by raw `Decimal`
**What goes wrong:** Passing a Prisma query result containing `Decimal` fields directly as a prop to a `"use client"` component throws at runtime (or produces a Next.js dev-overlay warning), not at compile time — TypeScript does not catch this.
**Why it happens:** `Decimal` is a class instance with methods (`toNumber`, `toString`, `toJSON`, etc.) and internal symbol properties (`nodejs.util.inspect.custom`); RSC serialization only accepts plain JSON-shaped data.
**How to avoid:** Map every Prisma result that includes `unitPrice`/`totalAmount` through an explicit `.toNumber()`/`.toString()` conversion inside the Server Component before rendering the Client Component (Pattern 2 above). This is the same class of bug already hit once in this codebase (Phase 1, Lucide icon components) — the fix pattern (convert to a plain value before crossing the boundary) is identical in spirit.
**Warning signs:** Dev console error "Only plain objects can be passed to Client Components from Server Components. Objects with symbol properties like nodejs.util.inspect.custom are not supported." — this is the exact string to grep for during manual verification.

### Pitfall 2: `poNumber` sequence gaps are expected, not a bug
**What goes wrong:** A developer sees `PO-0001`, `PO-0003` (missing `PO-0002`) after a failed Draft creation and assumes data corruption or a bug in the autoincrement logic.
**Why it happens:** PostgreSQL's `nextval()` on a `SEQUENCE` (which backs Prisma's `@default(autoincrement())`) is **never rolled back**, even if the surrounding transaction aborts — this is an intentional Postgres design tradeoff for sequence performance under concurrent inserts.
**How to avoid:** Document this explicitly as expected behavior (in code comments and/or a UAT note) so it isn't mistaken for a defect during verification. Do not attempt to "fill gaps" or re-sequence — that reintroduces race conditions.
**Warning signs:** None needed in code — this is a documentation/expectation issue, not a defect to guard against.

### Pitfall 3: `zodResolver` type mismatch requiring `as any` cast
**What goes wrong:** TypeScript complains about a type mismatch between the Zod-inferred type (with `z.preprocess` numeric coercion) and RHF's expected resolver type.
**Why it happens:** Documented codebase precedent — `zodResolver(schema) as any` is already the established workaround in `stock-client.tsx` and `products-client.tsx` for this exact issue (STATE.md: "`[Phase ?]: [03-02]: zodResolver(schema) as any cast applied to stock-client.tsx useForm calls to resolve z.preprocess/RHF type mismatch, matching existing products-client.tsx convention`").
**How to avoid:** Apply the same `as any` cast on the PO form's `useForm` call — do not attempt a "cleaner" typed fix that deviates from the established codebase convention (it will only create inconsistency, not correctness gain, given the same Zod version is in use).
**Warning signs:** TS2322-class error on `resolver:` prop when `quantity`/`unitPrice` fields use `z.preprocess`.

### Pitfall 4: Confirming a Draft with a deactivated supplier or product (D-16 stale-reference race)
**What goes wrong:** A Draft PO references a supplier or product that gets deactivated (`isActive: false`) by another user between Draft creation and confirmation; without a re-check, the PO silently advances to Ordered referencing an inactive entity.
**Why it happens:** The Draft may sit open for an arbitrary amount of time; `isActive` state is mutable independently of the PO.
**How to avoid:** `confirmPurchaseOrder` must re-query `supplier.isActive` and every line item's `product.isActive` at confirm time (not rely on the state captured at Draft-creation time) and reject with a message naming the specific deactivated entity (D-16).
**Warning signs:** Confirm succeeds even though a Product/Supplier "Deactivate" toggle was clicked in another tab during Draft editing.

### Pitfall 5: Double-receipt race if the PO row lock is skipped or ordered incorrectly
**What goes wrong:** Two concurrent receive requests (double-click, retry, two staff members) both read `status === "ORDERED"` before either commits, both proceed to create `StockTransaction` rows and increment `Product.currentStock` — physical shipment received once, but recorded twice.
**Why it happens:** Without `SELECT ... FOR UPDATE` on the `PurchaseOrder` row, there is no serialization point between the two concurrent transactions' status reads.
**How to avoid:** `tx.$queryRaw` `SELECT ... FOR UPDATE` on the `PurchaseOrder` row must be the *first* operation inside `prisma.$transaction`, and the `status === "ORDERED"` check must happen strictly after that query returns, before any `StockTransaction` create or `Product` update (D-22, Pattern 3 above).
**Warning signs:** Duplicate `StockTransaction` rows with identical `purchaseOrderId`/`productId`/`createdAt` (within milliseconds) in the audit trail; `Product.currentStock` off by exactly one shipment's worth.

## Open Questions

1. **Should `receivedQuantity` be allowed to be entered as `0` for a line item (short/failed shipment for that specific product) while other lines in the same PO receive normally?**
   - What we know: D-10/D-11 establish "one action, no partial-receipt state" at the PO level; D-11 allows staff to *edit* the received quantity per line for discrepancies.
   - What's unclear: Whether `0` is a valid edited value (meaning "this line item was not delivered at all, but the rest of the shipment was") or whether Zod should enforce `min(1)` per line (mirroring the existing `quantityField` pattern in `stock-transaction.ts`).
   - Recommendation: Allow `receivedQuantity >= 0` per line (not `>= 1`) — a `0` correctly represents a real-world short-shipment-to-zero-for-that-item scenario without requiring the "no partial receipt" state machine CONTEXT.md explicitly ruled out. The planner should make this an explicit Zod schema decision in `lib/validations/purchase-order.ts` rather than leaving it default-inherited from the stock-transaction pattern.
