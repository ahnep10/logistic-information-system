### Severity Tier Badge Colors (Phase 2 new)

Severity badges use the existing `badge.tsx` component (D-08) with custom `className` overrides. All three tiers use the **soft-tint pattern** (light background + dark text + border) to avoid visual aggression in data tables. Do NOT use the `destructive` variant for Critical — it uses a solid `bg-red-600` fill which is too harsh for inline table badges.

| Tier | Condition | className | Label |
|------|-----------|-----------|-------|
| Critical | `currentStock === 0` | `bg-red-100 text-red-700 border border-red-200 hover:bg-red-100` | `Critical` |
| Warning | `0 < currentStock <= reorderThreshold` | `bg-amber-100 text-amber-700 border border-amber-200 hover:bg-amber-100` | `Warning` |
| OK | `currentStock > reorderThreshold` | `bg-green-100 text-green-700 border border-green-200 hover:bg-green-100` | `OK` |

**Implementation pattern:**
```tsx
// Compute once per row — reuse this helper in Phase 3 (Warehouse) inventory screens
function getSeverityBadge(currentStock: number, reorderThreshold: number) {
  if (currentStock === 0)
    return { label: "Critical", className: "bg-red-100 text-red-700 border border-red-200 hover:bg-red-100" };
  if (currentStock <= reorderThreshold)
    return { label: "Warning", className: "bg-amber-100 text-amber-700 border border-amber-200 hover:bg-amber-100" };
  return { label: "OK", className: "bg-green-100 text-green-700 border border-green-200 hover:bg-green-100" };
}

// Usage in table row:
const severity = getSeverityBadge(product.currentStock, product.reorderThreshold);
<Badge className={severity.className}>{severity.label}</Badge>
```

**Phase 3 contract:** Extract `getSeverityBadge` to `lib/utils/severity.ts` so both Phase 2 (Products page) and Phase 3 (Warehouse inventory screens) import from the same source. Do not duplicate the logic.


**Why Tabs over Select for supplier filter:** A 3-state filter (All / Active / Inactive) with constant visibility is better served by Tabs than a Select dropdown. Tabs display all options at once, reduce click overhead, and make the current filter state immediately visible. Select would require an extra click to see the options.

**Filter Tabs specification:**
```
<Tabs defaultValue="all">
  <TabsList>
    <TabsTrigger value="all">All</TabsTrigger>
    <TabsTrigger value="active">Active</TabsTrigger>
    <TabsTrigger value="inactive">Inactive</TabsTrigger>
  </TabsList>
</Tabs>
```

Filter behavior:
- "All" tab: show all suppliers regardless of `isActive`
- "Active" tab: show only `isActive: true` suppliers
- "Inactive" tab: show only `isActive: false` suppliers

- Default on page load: "All" tab active

## Interaction Contracts

### Form submission pattern (all dialogs — identical to Phase 1)

1. User fills form and clicks submit.
2. Client-side Zod validation runs — invalid fields show `FormMessage` inline below each field.
3. If client valid: submit button enters Loading state, Server Action invoked.
4. Server re-validates with same Zod schema (`lib/validations/category.ts`, `product.ts`, `supplier.ts`).
5. On server error (e.g. SKU already exists, duplicate category name): return `{ error: "..." }`, render as `FormMessage` on the relevant field.
6. On success: `revalidatePath()` busts the page cache → dialog closes → table refreshes with new data.

### Duplicate name / SKU error rendering

- Category name duplicate: `FormMessage` below the Name field — "Category name already exists."
- SKU duplicate: `FormMessage` below the SKU field — "SKU already exists." (D-13 — exact copy required)
- Both are field-level errors, not page-level banners or toasts.

### Dialog open/close pattern (inherited from Phase 1 users-client.tsx)

- Dialog opens when user clicks "Create [entity]" or Edit icon button.
- "Discard" / "Discard changes" closes the dialog without submitting — discards any unsaved input.
- Escape key closes the dialog (Radix Dialog built-in behavior).
- Dialog close on success: close programmatically after `revalidatePath()` completes.
- Form state resets on close — do not persist partially-filled create form across open/close cycles.

### AlertDialog (deactivate/reactivate) pattern

- Clicking Deactivate/Reactivate icon button opens AlertDialog (not Dialog).
- "Keep active" / "Cancel" closes AlertDialog without action.
- Confirm action button enters Loading state while Server Action executes.
- On success: AlertDialog closes, table row updates to reflect new `isActive` state.

### Supplier filter behavior

- Filter is client-side state (`useState`) — selecting a tab instantly re-filters the displayed rows without a server request.
- Data is fetched once on page load (all suppliers); client component filters the array.
- If the supplier list is expected to grow beyond ~200 records in this MVP scope: server-side filter via URL param is acceptable, but not required for Phase 2.
- Default filter: "All" tab active on every page load.

### Category dropdown in product form

- Select is populated with `categories.filter(c => c.isActive)` fetched alongside product data in the server component via `Promise.all`.
- If zero active categories exist: Select component is `disabled`, and the inline message "No active categories — create a category first" renders below it as `text-xs text-zinc-500` (not a FormMessage — it is informational, not a validation error).
- If a product being edited has a now-deactivated category: show that category as a disabled option in the Select with a "(inactive)" suffix, e.g. "Old Category (inactive)". Do not allow saving with an inactive category — require the manager to select an active category before saving.

### Severity tier computation

- Computed in `lib/utils/severity.ts` — `getSeverityBadge(currentStock, reorderThreshold)`.
- Computed at render time from data returned by Prisma — no separate API call.
- Displayed as a static Badge in the Products table. No tooltip or expansion.
- Phase 3 (Warehouse) imports `getSeverityBadge` from the same file — do not duplicate.

### Role guard (server-side — all three entities)

- All create, update, and deactivate Server Actions call `requireManager()` before Prisma mutation.
- If a STAFF user calls a mutation directly (e.g. via API), the server returns a 403 error.
- UI hides action buttons for STAFF but that is not the security layer.
