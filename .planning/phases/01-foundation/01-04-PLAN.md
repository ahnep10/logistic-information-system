    STEP 2 — Create actions/users.ts:
    First line MUST be "use server" directive.
    Imports: auth from "@/lib/auth"; prisma from "@/lib/prisma"; hash, compare from "bcryptjs"; all three schemas from "@/lib/validations/user"; revalidatePath from "next/cache".

    Define async helper requireManager(): calls const session = await auth(). If session?.user?.role !== "MANAGER", throw new Error("Unauthorized — Manager role required"). Returns session. This is the inner gate — middleware is the outer gate; both must pass for manager operations.

    Export async function createUser(formData: FormData):
      Call await requireManager().
      Parse formData fields: name, email, role, password using createUserSchema.safeParse.
      If !parsed.success, return { error: "Invalid input. Please check all fields." }.
      Check for email uniqueness: const existing = await prisma.user.findUnique({ where: { email: parsed.data.email } }). If existing, return { error: "Email already in use." }.
      Hash password: const passwordHash = await hash(parsed.data.password, 12) — cost factor 12 per T-1-04.
      Create user: await prisma.user.create({ data: { name, email, role, passwordHash } }).
      Call revalidatePath("/users") to refresh the Server Component user list.
      Return { success: true }.

    Export async function updateUser(formData: FormData):
      Call await requireManager().
      Parse formData using editUserSchema.safeParse. Return { error } on failure.
      Build update data object: { name, email, role }. If newPassword is present and non-empty, add passwordHash: await hash(newPassword, 12) to the update data.
      Call await prisma.user.update({ where: { id: parsed.data.id }, data: updateData }).
      Call revalidatePath("/users"). Return { success: true }.

    Export async function toggleUserActive(userId: string, isActive: boolean):
      Call await requireManager().
      Call await prisma.user.update({ where: { id: userId }, data: { isActive } }).
      Call revalidatePath("/users"). Return { success: true }.

    Export async function changeOwnPassword(formData: FormData):
      Requires any authenticated session — not MANAGER-only: const session = await auth(). If !session?.user?.id, throw new Error("Unauthenticated").
      Parse formData using changePasswordSchema.safeParse. If !parsed.success, return { error: parsed.error.flatten().fieldErrors }.
      Fetch current user: const user = await prisma.user.findUnique({ where: { id: session.user.id } }). If !user, return { error: "User not found." }.
      Verify current password: const valid = await compare(parsed.data.currentPassword, user.passwordHash). If !valid, return { error: { currentPassword: ["Current password is incorrect."] } }.
      Hash new password with cost factor 12; update DB.
      Return { success: true }.
  </action>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-1-02 | Elevation of Privilege | actions/users.ts — createUser, updateUser, toggleUserActive | high | mitigate | requireManager() re-checks session.user.role === "MANAGER" at the start of each mutating action. Middleware is the outer gate; Server Action is the inner gate. STAFF cannot reach this action even if they craft a direct fetch. |
| T-1-04 | Information Disclosure | actions/users.ts — createUser, updateUser | high | mitigate | Password hashed with bcryptjs hash(password, 12) before DB write. Raw password never stored or logged. |
| T-1-05 | Tampering | actions/users.ts — createUser | high | mitigate | role field is parsed through z.enum(["MANAGER", "STAFF"]) — any value not in this enum causes safeParse to fail and return { error }. Mass assignment via role spoofing is impossible. |
| T-1-PWD | Spoofing | changeOwnPassword — current password verification | high | mitigate | compare(currentPassword, user.passwordHash) via bcryptjs. If compare returns false, action returns { error } — no DB write occurs. Prevents unauthorized password reset by a logged-in attacker sharing a browser session. |
| T-1-SELF | Tampering | toggleUserActive — own account deactivation | medium | mitigate | UI disables the deactivate button when user.id === session?.user?.id. Server Action does not explicitly prevent this (the Manager role is still required), but UI friction prevents accidental self-lockout. |
</threat_model>
